## Layering Principles

| Layer | Directory | Lifetime | Versioned |
|---|---|---|---|
| Entrypoints | `AGENTS.md` / `README.md` / `ARCHITECTURE.md` | Long | Yes |
| Contracts | `.harness/schemas/` | Long | Yes |
| Samples | `.harness/templates/` | Long | Yes |
| Rules | `.harness/rules/` | Long | Yes |
| Skills | `.harness/skills/` | Long | Yes |
| Tools | `.harness/scripts/` | Long | Yes |
| Tests | `.harness/tests/` | Long | Yes |
| Runtime | `work/` | Short | Partial; `work/plans/*` and `work/sessions/*` are recommended, `workflow-state.json` is optional |
| Business code | Target-defined source roots | Independent | Yes |

Core invariant: **`.harness/` contains only contracts, templates, rules, skills, and tools; `work/` contains only data.** Clearing `work/` returns the repository to an initial runtime state without damaging Harness itself.

Harness fixes only the `.harness/` and `work/` boundaries. Business code directories are not governed by Harness framework constraints; target project architecture and `.harness/contracts/project-contracts.json` declare them through `projectProfile.sourceRoots`.

## Key Files

### Entrypoint Layer

- **`AGENTS.md`**: the Harness entrypoint and navigation source. Agents read it first, then follow the linked rules.
- **`README.md`**: human-facing overview, installation guide, and usage notes.
- **root `ARCHITECTURE.md`**: target project business architecture. It describes business modules, dependencies, data flow, runtime topology, and project boundaries. It does not carry Harness framework architecture.
- **`.harness/ARCHITECTURE.md`**: stable reference for Harness framework architecture. Target project agent entrypoints reference it through the managed block.

### `.harness/schemas/`

Machine-checkable contracts. All schemas use Draft 2020-12.

- `workflow-state.schema.json`: structure and cross-field consistency for current workflow runtime state.
- `tasks.schema.json`: structure for a plan's internal task list, including compact verification and review gate summaries.
- `backlogs.schema.json`: intake-side pending backlog queue contract. It records incoming work not yet taken over by a workflow or plan, and does not activate plans, activate tasks, or modify workflow state.
- `backlog-consumption-event.schema.json`: audit event contract written to `work/backlog/consumed.jsonl` when a backlog item is taken over by a downstream workflow or plan. It preserves the complete original item, consumption time, target reference, and reason.
- `project-contracts.schema.json`: project environment contract structure, including project profile, source roots, command registry, environment checks, severity, and adapter fallback metadata.
- `project-entrypoints.schema.json`: target project agent entrypoint contract, including canonical entry, detected entries, managed block state, and `.harness/ARCHITECTURE.md` reference.

### `.harness/templates/`

Initialization samples. JSON templates place `$schema` at the top and point to `.harness/schemas/` so editors can validate and complete them immediately. Markdown templates provide structured body shapes constrained by the corresponding skills and scripts.

`entrypoint-managed-block.template.md` is the versioned source text for the Harness managed block in target agent entrypoints. `init-project-entrypoint.py` only reads the template, validates markers/version, and performs deterministic replacement; it does not maintain long managed-block prose inside Python strings.

### `.harness/contracts/`

Project-level contract directory. The directory is versioned with Harness; `project-contracts.json may be absent until project-env-contract configures it`. When absent, `.harness/scripts/check-project-env.py` returns `NOT_CONFIGURED`, meaning the project environment contract has not been initialized, not that Harness core is damaged.

The default output for `project-env-contract` is `.harness/contracts/project-contracts.json`, which is the truth source for project environment checks. `.harness/scripts/check-project-env.py` may only read that contract and execute declared commands or probes; it must not freely infer project facts from the repository, and it must not replace `session-start.py`.

`project-entrypoints.json` is generated or updated by the `project-init` and `project-update` skills through `.harness/scripts/init-project-entrypoint.py`. It records deterministic entrypoint facts: canonical target project entrypoint, detected entries, managed block state, managed block version, root `ARCHITECTURE.md` business architecture reference, and `.harness/ARCHITECTURE.md` framework architecture reference.

`project-entrypoints.json` is deterministic entrypoint metadata, not a semantic conflict report. Workflow, task, testing, review, commit, handoff, backlog, or archive conflicts in target entrypoint prose are judged and reported by the Agent during `project-init`; they are not written into this contract. Absence means entrypoint contracts are not configured yet; it does not mean Harness core assets are broken.

### Target Agent Entrypoint Integration

`project-init` owns first target agent entrypoint integration. It is a workflow mapping layer: after reading existing target project `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, Copilot instructions, or editor rules, it maps that project's startup, planning, development, testing, review, commit, handoff, backlog, and archive semantics onto the Harness lifecycle instead of creating a second state machine in the target repository.

`project-update` owns already-onboarded target repositories after installer update; installer update owns fixed asset copying. It performs managed block synchronization and semantic conflict review for new Harness capabilities without rerunning first onboarding or rewriting user-owned prose outside the managed block.

Boundaries:

- Root `ARCHITECTURE.md` remains target project business architecture. It describes only target business modules, dependencies, data flow, runtime topology, and project boundaries.
- `.harness/ARCHITECTURE.md` remains Harness framework architecture. It describes only Harness layering, lifecycle, schemas, templates, rules, skills, scripts, and `work/` runtime boundaries.
- `.harness/contracts/project-entrypoints.json` stores only deterministic facts such as canonical entrypoint, detected entries, managed block state, and version.
- Semantic conflict review belongs in Agent output, session audit, handoff, or follow-up plans; it is not schema/script-checkable entrypoint contract data.
- `.harness/scripts/init-project-entrypoint.py` may only create or replace the `harness-engineering` managed block, ensure root `ARCHITECTURE.md` exists, and write the entrypoint contract. It must not parse free text to judge workflow conflicts.

### `.harness/rules/`

Rules contain only semantics schemas cannot express, such as "`nextAction` must be one atomic sentence" or "which artifacts and gateways a phase transition requires." Rules should not duplicate schema constraints.

- `task-level.md`: L0/L1/L2/L3 task level classification. This is the fixed distributable runtime rule asset; it must not depend on source-repository `harness-design/` notes.
- `backlog-rules.md`: backlog intake `queue` / `preempt` semantics, pending queue boundary, and consumption audit. Intake and consume do not directly change the active workflow.
- `handoff-rules.md`: L2/L3 active plan `handoff.md` structure, recovery summary boundary, and lifecycle transaction entry shape. `handoff.md` is not a truth source.
- `llm-script-boundary.md`: scripts handle deterministic, repeatable, auditable operations; LLMs handle semantic judgment, natural-language artifacts, closure, and exception analysis. Runtime skills do not depend on source-only learning notes.
- `session-start.md`: three session startup paths, first-state bootstrap boundary, existing state read-only constraint, and session audit semantics.
- `workflow-gates.md`: testing, review, Architecture Impact, commit, handoff, and archive are workflow gates or audit actions, not standalone tasks.

### `.harness/skills/`

Agent process layer for producing and maintaining Harness artifacts. Skills describe workflow, judgment criteria, and artifact boundaries. They do not store runtime data, replace schema validation, or execute deterministic operations that scripts should own.

`.harness/skills/` skills are repo-local by default and serve this Harness artifact system. They may borrow the general Agent skill format, but they are not promised to run independently of `.harness/` schemas, templates, rules, and scripts. Cross-repository reuse should first abstract dependency contracts before moving them into installable generic skills.

- **`.harness/skills/project-init/SKILL.md`**: top-level semantic skill for initializing Harness into a target development repository. It owns entrypoint discovery, Harness architecture reference placement, and subworkflow orchestration; it does not replace deterministic scripts.
- **`.harness/skills/project-update/SKILL.md`**: semantic skill for synchronizing already-onboarded target repositories after fixed asset updates. It requires installer update first, runs managed block synchronization through `init-project-entrypoint.py`, reports marker-outside recommendations for user-owned prose, and does not write workflow or task runtime state.
- **`.harness/skills/project-env-contract/SKILL.md`**: semantic skill for generating project environment contracts. It guides the Agent to read repository evidence before writing `.harness/contracts/project-contracts.json`, including project profile, environment checks, and command registry. Project environment differences belong in project contracts, not in `session-start.py`; `session-start.py` only validates Harness startup core assets and runtime shape.
- **`plan-writing/SKILL.md`**: turns a requirement, backlog item, or approved design into an L2/L3 active plan package. It records expected Architecture Impact, completes the planning-time `Plan Review Gate` with `Status: passed`, then uses `materialize-tasks.py` to generate `tasks.json`. It does not activate tasks or write `workflow-state.json`.
- **`task-review/SKILL.md`**: produces a structured review summary from implementation, plan/task acceptance, verification evidence, Architecture Impact, and diff. The summary is input to `update-task.py`; the skill does not directly write `tasks.json` or `workflow-state.json`.

### `.harness/scripts/`

- **`harness`**: unified CLI entrypoint. It normalizes arguments and dispatches thinly to existing scripts; it does not reimplement lifecycle logic. The `validate-state` subcommand validates `work/workflow-state.json` by default.
- **`session-start.py`**: session startup orchestrator. It checks key Harness artifacts, including repo-local core skills, checks environment availability, runs `lint-harness.py`, bootstraps the first `workflow-state.json` only when state is missing and no active plan exists, runs `validate-state.py`, and writes a `work/sessions/YYYY-MM-DD/session-<id>.md` audit snapshot. It must not modify existing state, activate tasks, or advance phase.
- **`validate-state.py`**: three-layer validation: JSON Schema, cross-file checks such as `activeTaskId in tasks.json`, and semantic heuristics.
- **`materialize-tasks.py`**: generates `tasks.json` from the task contract block in a `plan.md` that has passed `Plan Review Gate`. It validates schema, taskId, anchor, dependsOn existence and acyclicity, file boundaries, acceptance, and verification. It only writes `tasks.json` inside the plan directory, initializes `review.lastResult=not_run`, and does not activate tasks or write `workflow-state.json`.
- **`update-task.py`**: task state write gateway for `tasks.json`. It updates task `status`, `ownerRole`, `currentStep`, `nextAction`, `verification`, `review`, and `blockedReason`, then validates `tasks.schema.json` and done preconditions.
- **`select-next-task.py`**: read-only selector. It validates the plan's `tasks.json`, selects the first `idle` task whose dependencies are all `done` when there is no active task, or emits a state patch suggestion to enter `archiving` when every task is done. It writes neither `tasks.json` nor `workflow-state.json`.
- **`state-write.py`**: the only update gateway for `workflow-state.json`. It receives JSON Patch or explicit fields, reads current state, applies the patch, validates phase transition path, workflowId immutability, terminal reset / terminal close, and key preconditions, runs `validate-state`, preflights change-log writability, atomically writes via temp file and rename, then appends a change log. For `reviewing -> archiving`, it reads the active plan's `tasks.json` to confirm the active task and all plan tasks are `done`. Re-entering `active` from `completed` or `archived` must use explicit terminal reset and a new `workflowId`. Planned terminal reset also requires a passed `Plan Review Gate`. Closing `active` into `completed` or `archived` must explicitly use `--allow-terminal-close` and leave no active plan directory. Except for the first-state bootstrap performed by `session-start.py`, every other script must output patches rather than directly writing state.
- **`start-workflow.py`**: starts a new workflow only from `completed` or `archived` terminal state. Direct L0/L1 workflows enter `implementing/developer`; planned L2/L3 workflows bind an existing active plan package and enter `planning/planner`. It dry-runs in an isolated copy, then writes real state through `state-write.py --allow-terminal-reset` and runs lint/validate postflight.
- **`lifecycle-transaction.py`**: lifecycle transition coordinator. For one transition, it performs preflight, isolated dry-run, calls `update-task.py` and `state-write.py`, appends `handoff.md`, and runs postflight. It does not bypass the lower-level gateways. Current actions include `activate-next`, `start-testing`, `start-review`, `review-failed`, and `review-passed`; review transitions consume the structured review gate stored in `tasks.json`.
- **`commit-task.py`**: L2/L3 task completion Git commit gate. It runs only after `lifecycle-transaction.py review-passed`, confirms the target task is `done` and both verification/review passed, then runs `git add -A` and `git commit`. The same commit may include the next task activation state produced by `review-passed`. It does not write `workflow-state.json` or `tasks.json`, and it does not replace lifecycle transition logic.
- **`archive-plan.py`**: archive tool. It requires `currentPhase=archiving`, all active plan tasks `done`, and an Agent-written `closure.md` containing `Architecture Impact`. It locates `.harness/` and `work/` from the Harness root, allows the Git top-level to be a parent of that root, confirms through the Git worktree that the task completion commit gate is closed, and allows only the current plan's `closure.md` to be uncommitted before archive. It then migrates the active plan package to `work/plans/archived/<PLAN-ID>/` and closes workflow state through `state-write.py --allow-terminal-close`.
- **`complete-workflow.py`**: L0/L1 direct workflow closeout tool. It requires no active plan, no active task, and `reviewing/reviewer`; the caller must provide verification evidence, review summary, and architecture impact summary. It preflights completion audit writability, closes workflow state through `state-write.py --allow-terminal-close`, and atomically appends a record to `work/sessions/YYYY-MM-DD/workflow-completions.jsonl`.
- **`backlog-intake.py`**: backlog intake write gateway. It initializes missing `work/backlog/backlogs.json` from `.harness/templates/backlogs.template.json`, allocates `BL-NNN` from `nextId`, increments the cursor, migrates old stores without `nextId` using `max(existing BL-NNN) + 1`, validates the full store, and atomically appends. It does not write `workflow-state.json`, `tasks.json`, or active plan files. `preempt` only requests LLM evaluation; it does not automatically interrupt the current workflow.
- **`backlog-consume.py`**: backlog consumption write gateway. It removes an item from `work/backlog/backlogs.json` only after downstream ownership evidence exists, and appends a schema-valid event to `work/backlog/consumed.jsonl`. A `plan:<PLAN-ID>` target requires a complete active plan package, passed `Plan Review Gate`, valid `tasks.json`, and plan/handoff reference to the backlog id or `sourceRef`. A `workflow:<workflowId>` target requires validated direct workflow state and session audit source reference. If the consumed event is written but pending store write-back fails, the script rolls back the event so the item is not present in both pending and consumed audit. It does not write `workflow-state.json`, `tasks.json`, active plan files, or `handoff.md`.
- **`check-project-env.py`**: project environment contract runner. It validates `.harness/contracts/project-contracts.json` or a caller-provided contract against `.harness/schemas/project-contracts.schema.json`, validates command/check ID uniqueness and `commandRef` references, then executes declared commands or probes. Contracts are the truth source; the runner does not write `workflow-state.json`, does not write `tasks.json`, and does not run automatically inside `session-start.py`.
- **`init-project-entrypoint.py`**: target project agent entrypoint runner. It detects `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, and similar entrypoints, returns `NEEDS_ENTRYPOINT` when none exists, and creates or updates entrypoint references only through the single `harness-engineering` managed block. Managed block content comes from `.harness/templates/entrypoint-managed-block.template.md`; the script validates marker/version and performs deterministic replacement. Multiple or unpaired managed block markers block execution to avoid duplicate entrypoint instructions. When writing and root `ARCHITECTURE.md` is missing, it creates an empty file for later task completion architecture review. It then writes `.harness/contracts/project-entrypoints.json`. It does not write workflow/task runtime state.
- **`lint-harness.py`**: read-only structure and invariant inspection. It covers initial `work/` shape, single active plan, active plan package completeness, active `handoff.md` structure, `activePlanRef` and directory consistency, active task count, and direct writes to `workflow-state.json` in production scripts under `.harness/scripts/`.

### `.harness/tests/`

- **`test_*.py`**: regression tests for Harness contracts, scripts, and templates. Tests and production scripts live in separate directories so `.harness/scripts/` does not serve both as tool entrypoints and test collection.

Planned lifecycle tool:

- **`check-env.py`**: checks dependencies such as `python`, `jsonschema`, and `git`. Failures do not block automatically; the report goes to the Agent for judgment.

### `work/`

- **`workflow-state.json`**: carries only runtime state; see `workflow-state.schema.json` and the rules.
- **`backlog/backlogs.json`**: pending queue for backlog intake, constrained by `.harness/schemas/backlogs.schema.json` and initialized from `.harness/templates/backlogs.template.json`. It stores only incoming work that has not yet been taken over by a workflow or plan; it is not an active plan and does not drive current workflow phase.
- **`backlog/consumed.jsonl`**: backlog consumption audit log. Each line is constrained by `.harness/schemas/backlog-consumption-event.schema.json` and records the full original item, consumption time, target reference, and reason.
- **`plans/active/<PLAN-ID>/`** and **`plans/archived/<PLAN-ID>/`**: symmetric active/archive plan paths; archiving changes only the active/archive segment.
- **`sessions/YYYY-MM-DD/session-<id>.md`**: session startup and Agent semantic audit evidence. It is evidence, not a workflow/task truth source.
- **`sessions/YYYY-MM-DD/workflow-completions.jsonl`**: L0/L1 direct workflow completion audit records, appended by `complete-workflow.py`, containing verification evidence, review summary, and architecture impact summary.

### Project Code Roots

Business code roots are target-defined. They may be a single `src/`, a .NET solution, a Java multi-module repository, or multiple monorepo roots such as `apps/`, `services/`, `packages/`, or `modules/`. Harness does not prescribe business directory names; it only requires business code not to cross into `.harness/` framework assets or `work/` runtime state.

Deterministic project facts are written to `.harness/contracts/project-contracts.json` under `projectProfile.sourceRoots`. Business modules, dependencies, data flow, runtime topology, and project boundaries remain described by root `ARCHITECTURE.md`.

## Key Invariants

1. **Single active plan**: `work/plans/active/` may contain at most one directory at any time. Rules and `lint-harness.py` enforce this.
2. **Schema first**: any constraint expressible by schema must live in `.harness/schemas/`; `.harness/rules/` must not duplicate it.
3. **Symmetric paths**: active and archived plan paths differ by one segment, `active` versus `archived`, allowing mechanical migration.
4. **Unified entrypoint**: ordinary external calls go through `.harness/scripts/harness <subcmd>`; lower-level scripts remain directly runnable for tests, debugging, and orchestration.
5. **Clearable runtime**: `rm -rf work/` returns to initial runtime state without damaging Harness itself.
6. **Single state writer**: existing `workflow-state.json` is updated only by `state-write.py`; `session-start.py` may create the first state only when state is missing and no active plan exists. Any other script that needs a state change must output a patch and let `state-write.py` write it. `lint-harness.py` scans production Python and extensionless scripts under `.harness/scripts/` for direct write patterns to that file and treats them as violations.
7. **Commit immediately after task completion**: after each L2/L3 task enters `done` through `review-passed`, `commit-task.py --task <TASK-ID>` must create a Git commit before the next implementation or archive closure starts. Commit gate is not a task; it does not change state or tasks, and it audits the completed task boundary.

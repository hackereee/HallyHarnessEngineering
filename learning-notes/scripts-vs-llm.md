# Scripts vs LLM：如何在 Harness 中分工

> 场景：Harness Engineering 设计中，当模型足够强（如 Claude Opus 4.7），是否可以摒弃脚本、全部交给 LLM 编排？

## 结论

**不应该摒弃。** 模型能力强 ≠ 用 LLM 替代脚本合理。要把"能不能"和"应不应该"分开。

---

## 一、脚本不可替代的五条硬边界（与模型能力无关）

| 边界 | 原因 |
|---|---|
| **成本 / 延迟** | LLM 调用 = 秒级 + 分钱；脚本 = 毫秒 + 零成本。高频校验不能靠 LLM |
| **确定性** | 同输入同输出；LLM 即使 `temperature=0` 也会因批次 / 模型版本漂移 |
| **可审计** | 脚本能精确回溯到规则链；LLM 只能给"我判断是……" |
| **可测试 / 可回归** | 脚本能写单测锁住行为；LLM 判断无法 diff、无法回归 |
| **安全边界** | 脚本可审计可冻结；LLM 可被 prompt injection 绕过 |

任何一条出问题都是**系统性事故**，不是"模型更聪明"能修的。

---

## 二、LLM 无可替代的三件事

| 场景 | 脚本为何不行 |
|---|---|
| **语义判断** | "nextAction 是否真的原子" / "handoff 是否隐瞒风险"——标准模糊，无法形式化 |
| **未预期输入** | 脚本遇到 shape 不对就 crash；LLM 能 degrade gracefully |
| **自然语言产出** | plan 正文、closure 复盘、handoff 正文——必须 LLM 写，脚本只管壳 |

---

## 三、判断框架：grep 能否代替 LLM

问三个问题，**全为"否"**才让 LLM 上：

1. 能用正则 / AST / schema 表达吗？能 → 脚本
2. 输入空间是否有限可枚举？是 → 脚本
3. 错误是否可逆？不可逆（删文件、改 git、发 PR）→ **一定**脚本 + LLM 二次确认

---

## 四、可靠性复利：端到端可靠性是相乘的

10 步工作流：

| 分工 | 端到端可靠性 |
|---|---|
| 全脚本（每步 99.9%） | ≈ 99% |
| 全 LLM（每步 98%） | ≈ 82% |
| 混合（8 脚本 + 2 LLM） | ≈ 95% |

**每一个"LLM 替代脚本"的决定都在砍可靠性预算。**

---

## 五、Claude Code 的启示

Claude Code 本身是一个**大的确定性系统**（CLI、hooks 引擎、schema 校验、权限系统、文件系统抽象），LLM 只是其中一个组件。工具调用、权限检查、文件读写、hooks 分发全是代码。

如果连"高级模型驱动的 agent"都大量脚本化，更没理由为了省脚本而承担那五条硬边界风险。

---

## 六、Harness Engineering 的具体分工

### 脚本负责（硬约束 + 可形式化）

- JSON Schema 结构、枚举、跨字段一致性
- 路径迁移、原子文件操作（`active` ↔ `archived`）
- 单活跃 plan、目录完整性（`lint-harness.py`）
- git 状态检查、权限 gate
- 会话 ID / 时间戳 / 路径生成

### LLM 负责（语义 + 创造）

- 写 plan / tasks / handoff / closure 正文
- 判断 `nextAction` 是否原子、task 是否真的完成
- 从 backlog 晋升哪条到 plan（优先级判断）
- 失败时提出修复建议
- 审读对方 Agent 的 handoff 质量

### 关键动作双签（脚本 + LLM 联合）

| 动作 | 脚本部分 | LLM 部分 |
|---|---|---|
| `archive-plan` | 原子迁移目录 | 写 `closure.md`，复查是否遗漏 open question |
| `select-next-task` | 筛出合法候选集 | 在候选里做语义优先级排序 |
| state 变更 | `validate-state.py` 通过 | 复查"是否真的反映了刚做的事" |

---

## 七、一句话心法

> **脚本是骨架，LLM 是肌肉。没有骨架的肌肉瘫软，没有肌肉的骨架僵死。**

模型越强，越应该把**高价值判断**交给它，而不是让它做 `validate_state` 这种跑一万次都一样的累活。**摒弃脚本是在浪费模型能力，不是在发挥它。**
